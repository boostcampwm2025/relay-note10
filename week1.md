## 토론 내용 (팀원분들의 말을 모두 정리했습니다~)

### 1. 부스트캠프 커뮤니티는 어떤 커뮤니티가 되어야 할까? 커뮤니티라는 건 무엇일까?

솔직히, “좋은 사람들끼리 덜 눈치 보면서, 자유롭게 질문하고 도움받을 수 있는 곳”이면 충분하다고 생각해요. 커뮤니티가 괜히 뭔가 잘난 척하는 분위기로 흐르면, 오히려 질문하기 부담스러워지고, 실수나 모르는 걸 드러내는 게 망설여질 때가 있잖아요. 오히려 같이 어설프게 삽질도 하고, 모르는 건 서로 응원해주고, 각자 자기 속도대로 배워도 괜찮은 분위기였으면 해요.
잘하는 사람들이 모인 곳이 아니라, 계속 발전하는 사람들이 모인 커뮤니티였으면 좋겠다는 의견도 있었어요.
부스트캠프는 결국 개발자를 꿈꾸는 사람들이 모인 곳이니까, 혼자 고민하는 것보다 함께 고민하고, 함께 성장하는 공간이 되었으면 해요. 다양한 배경과 관점을 가진 사람들이 모이니, 서로의 새로운 시각에서 자극받는 게 진짜 커뮤니티의 힘인 것 같아요.

### 2. 개발자로서 학습하고 성장한다는 건 무엇일까? 학습과 성장이 되고 있는지 어떻게 확인할 수 있을까?

코드를 많이 짠다고 실력이 쌓인다고 생각하지 않아요.
문제를 진짜로 이해하고, 무엇을 만들어야 할지 스스로 파악하고, 논리적으로 설계해서 구현까지 해내는 힘, 그게 진짜 개발자로서의 성장인 것 같아요.
하지만 혼자서 ‘내가 제대로 성장하고 있나?’를 확인하기가 참 애매하죠. 그래서 피어 피드백이 도움이 많이 돼요. 내가 놓친 게 뭔지, 어떻게 더 생각할 수 있을지 알려주기도 하고.
사실 매일 주어지는 미션들을 고민하고, 또 서로 토론하면서 해결하는 그 과정 하나하나가 모두 성장의 순간이라고 생각해요. 앞으로는 AI도 또 다른 동료처럼, 내 성장을 점검해주는 파트너로 더 잘 활용해보고 싶어요.

### 3. AI에게는 어떤 도움을 받을 수 있고, 받을 수 없을까? AI가 할 수 있지만 하면 안 되는 일도 있을까?

AI에게서 도움받을 수 있는 부분이 분명 있죠. 예를 들어, 내가 설계나 구현을 하다가 뭔가 모호한 부분이 있을 때, 관련된 CS 지식을 짚어준다든지, 내가 놓치고 있는 포인트를 피드백해주는 역할은 정말 유용하다고 생각해요.
하지만 중요한 건, 코드 작성이나 설계 자체를 AI에게 넘겨주면 안 된다는 거예요. 고민도 해보기 전에 바로 AI에게 묻는다면 내 사고력이나 창의성이 키워질 틈이 없겠죠.
AI는 어디까지나 ‘수단’이지, 성장의 ‘목적’이 되어선 안 된다고 생각해요. 나와 AI 사이에 선을 확실히 긋는 연습이 필요한 시점인 것 같아요.

### 4. 챌린지를 하며 놓치고 있는 것과 과도하게 집중하고 있는 것은 무엇일까? 어떻게 보완할 수 있을까?

이번 챌린지를 하면서 설계에서 구현으로 넘어갈 때 아직 부족함을 많이 느꼈어요. 반대로, CS 지식 정리에 시간 투자를 과하게 했던 것 같기도 하고요.
보완하려면, 설계 단계에서 다이어그램이나 시각화 같은 방법을 통해 내 생각을 정리해보고, 그걸 AI에게 피드백받는 식으로 접근하면 좋을 것 같아요.
그리고 CS 지식도 꼭 필요한 포인트만 AI에게 추천받아서, 정말 구현에 바로 적용할 수 있을 때만 깊이 파는 식으로 효율을 높이면 좋겠다는 생각이 들어요.

### 5. 부스트캠프 1주차를 보내면서 아쉬웠던 점, 시도해보고 싶었지만 망설였던 게 있나?

사실 1주차를 보내며, 내가 스스로 구현을 해내는 힘이 아직 부족하다는 점이 제일 아쉬웠어요.
어떤 기능을 만들지, 어떻게 설계해야 할지 고민은 많이 했지만, 직접 코드를 써서 해결하는 부분에서는 자꾸 머뭇거렸던 것 같아요.
그래도 설계 단계에서 AI에게 방향성을 점검받으면서 그 과정을 학습한다면, 점차 구현력도 자연스럽게 따라오지 않을까 싶어요. 앞으로는 일단 부딪혀보면서, 부족하더라도 계속 시도하는 쪽을 택하려고 해요.

- 지식의 점수화: "리눅스 커널"이나 "VFS" 같은 거대 담론보다는, 내 코드에 직접 적용할 핵심 개념 위주로 지식을 뾰족하게 다듬을 필요가 있습니다.

### 설계가 구체화되면, 이제 구현에 딱 필요한 CS 지식만 골라서 학습하는 겁니다.

User의 질문:

- "재귀적으로 경로를 탐색하는 함수를 만들어야겠네. 그리고 exit할 때 객체 상태를 파일에 저장해야 하는데, 이때 필요한 CS 개념이나 Node.js 기술이 있을까?"

### 🤖 AI의 피드백 (핵심 지식 추천):

네, 정확합니다! 지금 단계에서는 두 가지에 집중하면 좋습니다.

- 자료구조 (자바스크립트 Map): 디렉토리의 자식 목록을 관리할 때 배열이나 일반 객체보다 Map을 사용하면 항목 추가, 삭제, 조회가 훨씬 효율적입니다.

- 알고리즘 (재귀 Recursion): 경로 탐색처럼 깊이가 정해지지 않은 트리 구조를 탐색할 때 재귀 함수는 가장 직관적이고 강력한 도구입니다.

- 직렬화 (Serialization): 메모리에 있는 객체를 디스크에 저장 가능한 문자열이나 바이트 스트림으로 변환하는 과정입니다. Node.js에서는 JSON.stringify()가 가장 손쉬운 직렬화 방법이며, fs.writeFileSync로 파일에 쓰면 됩니다. 반대로 파일에서 읽어 객체로 되살리는 것은 JSON.parse()(역직렬화)를 사용합니다.

이런 방식으로 [요구사항 정의 → 설계 시각화 → AI 피드백 → 핵심 지식 학습 → 구현] 사이클을 반복하면, CS 지식이 공중에 뜨지 않고 내 코드에 바로 녹아드는 경험을 하실 수 있을 겁니다. 지금의 학습 방식과 고민은 매우 올바른 방향으로 나아가고 있다는 증거이니, 자신감을 갖고 도전해 보세요!

### 👍 잘하고 있는 점 (과도하게 집중하고 있는 부분)

- 개념의 본질을 파고드는 탐구력: 단순히 '프로세스와 스레드는 다르다'에서 그치지 않고, '왜 스레드는 스택과 PC 레지스터를 독립적으로 가져야 하는가?'를 요리사, 작업대, 책갈피 같은 훌륭한 비유로 설명하며 완전히 자기 것으로 만들었습니다. 이는 복잡한 CS 지식을 제대로 소화하고 있다는 강력한 증거입니다.

- 용어의 명확한 구분: 병렬성(Parallelism)과 동시성(Concurrency)의 차이를 다이어그램과 함께 명확히 구분하고, "병렬성은 동시성이기 위한 필요조건이지만 충분조건은 아니다"라고 정리한 부분에서 깊은 이해도가 엿보입니다.

- 이론과 실제의 연결: 학습한 스레드 개념을 JavaScript 환경에 적용하여 '왜 JS는 싱글 스레드일까?', '이벤트 루프는 어떻게 동시성을 지원하는가?', 'Node.js에서 병렬 처리를 하려면 어떻게 해야 하는가?'까지 연결하며 지식을 확장하고 있습니다.

스스로 지적했듯이, 이 부분은 **'CS 지식 정리에 시간을 과하게 투자'**한 결과물이기도 합니다. 하지만 이 깊이 있는 이해가 앞으로 만들 코드의 품질을 결정하는 든든한 자산이 될 것입니다.

### 🤔 보완할 점 (놓치고 있는 것)

- 학습 내용에서 명확하게 드러나는 한 가지는, **'그래서 이 지식으로 무엇을 만들 것인가?'**에 대한 구체적인 설계가 아직 없다는 점입니다.

- worker_threads 사용법을 학습했지만, 어떤 문제를 해결하기 위해 이 워커 스레드를 사용할 것인지에 대한 구체적인 시나리오가 빠져 있습니다.

- 레이스 컨디션(Race Condition)의 위험성을 배웠지만, 내 코드에서 어떤 데이터가 공유 자원(Shared Resource)이 될 수 있고, 이를 어떻게 보호할지에 대한 설계적 고민으로 이어지지 않았습니다.

- 바로 이 지점이 **'설계에서 구현으로 넘어갈 때 느끼는 부족함'**의 원인입니다. 이제부터는 학습한 지식을 실제 코드로 변환하는 다리를 놓는 데 집중해 봅시다.

### AI의 피드백으로 핵심 CS 지식 점검하기

🤖 AI의 피드백 (예시):

훌륭한 설계입니다! 말씀하신 흐름은 worker_threads의 핵심 동작 방식을 정확히 이해하고 계십니다. 여기서 학습한 CS 지식을 다음과 같이 연결하고 보완할 수 있습니다.

- 자바스크립트의 비동기 처리 (이벤트 루프): 7번 요구사항("계산 중..." 메시지 출력)을 구현하는 것은 학습하신 이벤트 루프 덕분입니다. 워커가 병렬적으로 계산을 처리하는 동안, 메인 스레드의 이벤트 루프는 계속 돌면서 setInterval 같은 비동기 작업을 처리할 수 있습니다. 이는 "하나의 스레드로도 대규모 비동기 I/O 작업을 효율적으로 처리"하는 것을 직접 구현해보는 경험입니다.

- 데이터 통신 방식: 3번과 5번 단계에서 메인-워커 스레드 간 데이터 전송은 메시지 전달(Message Passing) 방식을 사용합니다. 이는 학습하신 레이스 컨디션 문제를 원천적으로 회피하는 안전한 방법입니다. 만약 데이터를 복사하지 않고 메모리를 직접 공유하는 SharedArrayBuffer를 썼다면, 정리하신 레이스 컨디션 문제가 발생할 수 있었을 겁니다.

- 병렬성 vs 동시성: 이 설계는 학습하신 개념을 완벽하게 구현하는 예제입니다. 워커 스레드가 별도의 코어에서 계산을 수행하는 것은 **병렬성(Parallelism)**이고, 메인 스레드가 계산을 기다리지 않고 주기적으로 메시지를 출력하는 것은 **동시성(Concurrency)**입니다.

## J290 - 코드에 마음을 담다

### 선택한 배경

`코드에 마음을 담다`라는 미션 문장을 보자마자,<br>제가 가장 최근까지 만들고 있던 프로젝트, `MINT` 언어가 떠올랐습니다.

> MINT = Minda's Interpreted Natural Tongue

`MINT`는 *코드는 자연처럼 피어나야 한다*는 철학을 바탕으로<br>
시적이고 감성적인 문법을 가진 언어를 직접 설계해본 개인 프로젝트입니다.<br>

파서까지는 구현을 마치고 인터프리터는 부스트캠프에 참가하면서
잠시 멈춘 상태지만, <br>언젠가 꼭 다시 완성하고 싶은 프로젝트예요.

가장 최근까지 진행한 작업인데다 개인 프로젝트 중<br>가장 많이 담긴 작업이라
이번 퀘스트 주제로 선택하게 되었습니다.

> [_MINT 언어 설계 문서_](https://github.com/mindaaaa/Dev-Journey/tree/main/CS/mint-interpreter)<br>[_MINT 프로젝트_](https://github.com/mindaaaa/mint-lang)

### 수행 결과

#### 🪴 "Thread Garden: 충돌의 정원에서"

```mint
plant garden = "shared memory"
plant petals = 16
plant breeze = "race condition"

sparkle "🌱 스레드가 정원에 피어납니다"

bloom (petals times) softly {
  petal thread() {
    plant read = garden
    softly wait (a little)
    plant write = read + 1
    sparkle "🌿 one petal touches " + write
  }
  thread()
}

breeze (garden is disturbed) softly {
  sparkle "🍂 a petal was lost in the wind"
}

petal end() {
  sparkle "🍃 total: " + petals * 100
  sparkle "🌙 memory holds: " + garden
  gift "thank you for growing with us"
}
```

#### 🌾 스레드의 정원에서

```plaintext
조용한 정원에
열여섯 갈래의 빛이 내렸다.

각기 다른 이름을 가진 스레드들은
한 조각의 기억을
서로 나누어 담으려 했고,

어느 누구도
양보하지 않았다.

그러나,
그들은 몰랐다.

함께 쓰는 그 기억은
햇살처럼 예민하고,
순간처럼 사라지기에

읽는 순간, 이미 늦어버린다는 것을.

꽃은 피어나고
데이터는 흐르며
숫자는 쌓이지만

그 사이사이
조용히 스러진 것들이 있었다.

누군가 남긴 숫자가,
다음 사람에 의해
덮어씌워졌다는 사실.

그 정원의 끝에서
우리는 요약을 본다.

그러나 아무도
울지 않았다.

그저 조용히,
다시 스레드를 심고,
조용히 다시 피워낸다.

“꽃은 매번 정확히 피어나지 않아도,
그 자체로 존재의 의미가 된다.”
```

### 시와 코드의 연관성

MINT 코드 시에서는 스레드를 `정원의 꽃잎(petal)`으로, <br>공유 메모리를 `garden(정원)`으로 은유하여 표현했습니다.

각 스레드가 메모리에 접근하며 충돌을 일으키는 과정을, <br>꽃잎이 바람 속에서 흔들리며 기억을 덮어쓰는 순간들로 풀어냈습니다.

시에서는 이 과정을 정적인 정원 안의 작은 소란으로 그려내며,<br>
레이스 컨디션의 본질—*읽는 순간 이미 늦은*을 시적으로 묘사하고,<br>
결국 그조차도 자연의 일부로 받아들이는 태도를 담고자 했습니다.

> 코드가 기술을 보여줬다면, 시는 그 안에 담긴 의미와 감정을 들려줍니다.
